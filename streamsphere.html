The Design Problem

Every major streaming platform supports viewers and creators at once, that part isn't novel. The actual problem was structural: design a schema where three roles with genuinely different data needs, billing for viewers, follower metrics for creators, platform-wide visibility for admins, can all sit on the same USER record without forcing a redesign every time one role's requirements changed. Get the specialization wrong early and every table after it inherits the mistake.

The Calls I Made
  • Pushed the team to lock requirements per role first, before any table existed, so the schema served three actual workflows instead of one generic "user" assumption
  • Chose disjoint specialization with shared keys over a single flat USER table, the harder design up front, but the one that doesn't duplicate data as roles grow
  • Called out the ACTORS junction-table issue as a documented fix rather than a quiet patch, because how a team catches its own design mistakes belongs in the deliverable, not edited out of it
  • Treated the MongoDB conversion as five separate decisions, not one default, since embedding everything would have been faster to write and wrong to ship
Requirements by Role

Scoped before any table was drawn, the same sequencing a PRD forces: know what each user needs to do before deciding how the data is shaped.

ViewerBrowse, subscribe, track viewing progress. Needs subscription tier, payment history, and a per-session completion log.
CreatorUpload and grow a channel. Needs follower count, verification status, and visibility into content performance.
AdminOversee the platform. Needs analytical access across every entity, not just their own data.

01 / The Scenario

One platform, three roles, one database that has to hold all of it together.

StreamSphere Inc. runs a streaming platform that's also a creator economy: viewers consume movies, TV shows, and short clips, while independent creators upload and monetize their own content. The database has to support both halves at once: subscription billing and viewing analytics for viewers, follower counts and content performance for creators, and platform-wide oversight for admins, without duplicating data across any of them.

02 / Conceptual Design

USER and CONTENT carry the whole schema on their backs.

The EERD centers on two supertypes. USER splits into Viewer, Creator, and Administrator through a disjoint, total specialization, every user has exactly one role, no exceptions. CONTENT splits into Movies, TV Shows, and Short Clips the same way, each with its own type-specific attributes layered on a shared core.

USER → Viewer / Creator / Administrator

Specialization is implemented with shared primary keys: VIEWER, CREATOR, and ADMINISTRATOR each use user_ID as both their own primary key and a foreign key back to USER. No role-specific column ever touches the base USER table.

CONTENT → Movies / TV Shows / Short Clips

Same pattern. Title, language, release date, and description live once in CONTENT. Duration and rating belong only to MOVIES; season-and-episode hierarchy and a self-referencing spin-off link belong only to TV_SHOWS.

TV Show Hierarchy

TV_SHOWS contains SEASONS, and each SEASON contains EPISODES. An episode can't exist without its parent season, and a season can't exist without its parent show, a strict three-level chain enforced through foreign keys at every level.

Actors, Resolved Properly

Actors appear across both movies and TV shows, a genuine many-to-many relationship on both sides. Resolved through two junction tables, MOVIE_ACTORS and TV_ACTORS, rather than direct foreign keys on the ACTORS table itself. More on why that mattered below.

Two multivalued attributes got pulled out into their own tables rather than crammed into a single column: a content item's genres go into CONTENT_GENRE (a content item can carry more than one), and a user's contact numbers go into USER_CONTACT. Keeping these atomic is what 1NF actually requires.
03 / Caught During Implementation

A design mistake that only showed up once we tried to populate it.

The first version of ACTORS had two foreign key columns sitting directly on the table: Movie_id and TV_show_id. It worked, right up until we tried to insert an actor who appears in more than one show. The design quietly assumed every actor appears in at most one movie and one TV show, which falls apart the moment a real cast list does.

What broke

An actor with two TV credits has nowhere to put the second one. The schema can only point each actor row at a single movie and a single show, full stop.

What we changed

Dropped both foreign keys off ACTORS with ALTER TABLE, then introduced MOVIE_ACTORS and TV_ACTORS as proper many-to-many junction tables. The migration is in the submitted SQL file as an explicit, visible step, not silently folded into the original schema.

-- migrate ACTORS to many-to-many junction tables
ALTER TABLE ACTORS DROP FOREIGN KEY fk_actor_movie;
ALTER TABLE ACTORS DROP FOREIGN KEY fk_actor_tvshow;
ALTER TABLE ACTORS DROP COLUMN Movie_id;
ALTER TABLE ACTORS DROP COLUMN TV_show_id;

CREATE TABLE MOVIE_ACTORS (
  content_ID INT NOT NULL,
  Actor_id INT NOT NULL,
  PRIMARY KEY (content_ID, Actor_id)
);

This is the kind of fix that's invisible in a polished final diagram and very visible the moment you try to load real data through it. I wanted it documented as its own step rather than quietly cleaned up, since the team's actual design process included getting this wrong first.

05 / Normalization

Every table, checked against three rules, not assumed.

Normalization isn't a formality you wave at, it's the thing that decides whether a value lives in exactly one place. Each of the 20 tables was checked against all three rules individually before the schema was called done.

RULE 01
1NF

Every attribute holds one atomic value, no repeating groups in any row. This is exactly why genres got their own table (CONTENT_GENRE) and contact numbers got theirs (USER_CONTACT): both are naturally multivalued, and cramming either into a single column would have broken atomicity immediately.

RULE 02
2NF

Only relevant for tables with composite keys, five of them here: CONTENT_GENRE, CONTENT_CONSUMPTION, WATCHED, MOVIE_ACTORS, TV_ACTORS. Four are pure junctions with no extra attributes, so there's nothing to check. CONTENT_CONSUMPTION's one extra column, consumed_at, depends on both key columns together, which is the definition of compliance.

RULE 03
3NF

No non-key attribute can be determined by another non-key attribute. Subscription pricing lives only in SUBSCRIPTION_PLAN, never copied onto VIEWER. Title and language live only in CONTENT, never duplicated into MOVIES, TV_SHOWS, or SHORT_CLIPS. Checked table by table, all twenty pass.

06 / SQL Queries

Ten queries. Three of them actually run the platform.

All ten queries together cover INNER JOIN, LEFT OUTER JOIN, aggregates with GROUP BY, HAVING, and correlated subqueries. These three were picked out specifically because they map to a real decision someone on the platform would make, not just a SQL feature being demonstrated.

Query 02 · Creator Follower Ranking with Content Volume

SELECT u.first_name, u.last_name, c.Followers,
  COUNT(co.content_ID) AS total_content
FROM USER u
INNER JOIN CREATOR c ON u.user_ID = c.Creator_ID
INNER JOIN CONTENT co ON c.Creator_ID = co.creator_ID
GROUP BY u.user_ID, u.first_name, u.last_name, c.Followers
ORDER BY c.Followers DESC;

Feeds the Creator Leaderboard. Who sees it: platform admins and the creator partnership team. Decision it supports: which creators qualify for verified status, homepage featuring, or partnership agreements, and which ones have a big following but low upload volume worth targeting with incentives.

Query 04 · Content Engagement: Average Completion Rate and View Count

SELECT co.title,
  ROUND(AVG(h.completion_percent), 2) AS avg_completion,
  COUNT(h.history_ID) AS total_views
FROM CONTENT co
INNER JOIN HISTORY h ON co.content_ID = h.content_ID
GROUP BY co.content_ID, co.title
ORDER BY avg_completion DESC;

Who sees it: the content operations team, and creators through their own analytics view. Decision it supports: low-completion titles (Code Breakers at 45%) can get flagged for editorial review or ranked lower in recommendations; high-completion titles (Neon Horizon and Shadow Protocol, both at 100%) get prioritized in Continue Watching.

Query 09 · Above-Average Rated TV Shows with Cast

SELECT co.title, ts.ratings,
  COUNT(ta.Actor_id) AS actor_count,
  GROUP_CONCAT(a.Name ORDER BY a.Name SEPARATOR ', ') AS cast
FROM CONTENT co
INNER JOIN TV_SHOWS ts ON co.content_ID = ts.content_ID
LEFT JOIN TV_ACTORS ta ON ts.content_ID = ta.content_ID
LEFT JOIN ACTORS a ON ta.Actor_id = a.Actor_id
WHERE ts.ratings > (SELECT AVG(ratings) FROM TV_SHOWS)
GROUP BY co.content_ID, co.title, ts.ratings
HAVING COUNT(ta.Actor_id) >= 1
ORDER BY ts.ratings DESC;

Combines a rating threshold with cast aggregation in one pass. Application: populates Trending TV and Featured Cast on the homepage. Who uses it: viewers browsing recommendations, and editors curating featured placements.

07 / MongoDB Conversion

Same data, a different question for every relationship.

A relational schema resolves every relationship the same way: a join. MongoDB doesn't, so each relationship has to be decided individually between embedding it inside the parent document or referencing it from a separate collection. We ran every one of these five decisions through the same three questions.

Question 01

Do we usually access the data together?

Question 02

Is the data mostly used within a small number of documents?

Question 03

Is the embedded data bounded in size?

TV_SHOWS ↔ SEASONS ↔ EPISODES

Embedded
Co-accessThe show detail page renders seasons and episodes in one view, always together.
Document scopeEach episode belongs to exactly one season, each season to exactly one show. Never shared.
BoundednessThe number of seasons and episodes per show is fixed at publication. Predictable size.

VIEWER ↔ SUBSCRIPTION_PLAN

Embedded
Co-accessChecked on every authenticated request for quality tier and stream-limit enforcement.
Document scopeExactly one active plan per viewer, never shared across documents.
BoundednessA single fixed-size sub-document. Its size never grows over time.

VIEWER ↔ HISTORY and PAYMENT

Referenced
Co-accessOnly pulled on the Watch History or Billing pages specifically, not on every profile load.
Document scopeSpans many sessions and billing cycles, not scoped to a small set.
BoundednessGrows indefinitely for the life of the account. Embedding would make VIEWER documents balloon over time.

CONTENT ↔ ACTORS

Referenced
Co-accessContent metadata is usually loaded without needing cast details at all.
Document scopeOne actor appears across many titles. Embedding would mean duplicating their profile into every one.
BoundednessAn actor's filmography grows as new content publishes. Open-ended, not fixed.

CONTENT ↔ CREATOR

Referenced
Co-accessCreator profile pages are visited independently of any specific content item.
Document scopeShared across every item that creator has ever uploaded.
BoundednessA creator's portfolio is open-ended and keeps growing with every new upload.

Sample Document: TV_SHOWS with Seasons Embedded

{
  "_id": ObjectId("content_11"),
  "title": "Code Breakers",
  "ratings": 8.4,
  "creator_ref": ObjectId("creator_6"),
  "actor_refs": [ObjectId("actor_1"), ObjectId("actor_3")],
  "seasons": [
    { "season_id": 1, "number": 1, "episodes": [
      { "episode_id": 1, "number": 1, "length": 45 },
      { "episode_id": 2, "number": 2, "length": 47 }
    ]},
    { "season_id": 2, "number": 2, "episodes": [
      { "episode_id": 3, "number": 1, "length": 42 }
    ]}
  ]
}

creator_ref and actor_refs stay as ObjectId references; seasons and episodes nest directly inside the document. Same entity, two different strategies, each justified by the three questions above.

09 / Where It Landed

A schema that holds up across two database paradigms.

The relational design isn't just normalized on paper, it's implemented, populated, and queried in MySQL, and it survives translation into a document model without losing any of its logic. That's the actual test of whether a schema is sound: does it still make sense once you're forced to represent it a second way.

Final State 20 Tables.
3NF Verified.

Every table checked individually against 1NF, 2NF, and 3NF. Ten SQL queries covering the full range of required constructs, three tied directly to decisions a creator partnerships team, content ops team, or homepage editor would actually make. Five MongoDB relationships, each resolved on its own merits rather than a blanket embed-everything or reference-everything rule. One real schema mistake, caught during implementation and fixed in the open rather than smoothed over in the final write-up.
